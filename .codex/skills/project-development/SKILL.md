# Project Development

每次接手都先确认当前仓库事实，再执行任务。

事实来自 `spec.md`、`AGENTS.md`、`README.md`、代码、测试、配置、git 状态、命令结果和 owner 明确输入；如果当前任务存在 `plan.md`，事实也来自 `plan.md`。

## 核心规则

- 先确认事实，再按任务需要写或更新 `plan.md`，然后实现。
- 不把未实现能力写成当前事实。
- 不交半成品。
- 架构、类型、测试、文档和验证必须作为同一次交付的一部分。
- 当前生产级目标高于历史形态。
- 需要重建时直接建立当前主干。
- 历史数据保留、旧接口保留和旧页面保留只在 owner 明确要求并写入当次 `plan.md` 后执行。

## 文件职责

- 单一职责看变化原因，不看行数。
- 超过 300 行必须触发职责审查，但不是自动拆分理由。
- 能一句话说清职责、变化原因一致、内部耦合合理，可以保留。
- 状态管理、规则计算、数据读写、渲染展示、外部接线、错误处理、业务判断混在一起时，必须拆清边界。
- 不为了拆而拆，不为了省事硬塞。

## 技术纪律

按项目技术栈填写：

- 接入层只做参数接入、鉴权上下文接入和响应转换。
- 业务层承载业务流程。
- 数据访问层承载存储读写。
- UI 层只做交互编排和展示。
- 复杂业务规则放入可测试模块。
- 接口必须有请求、响应、错误和权限边界。

## 测试纪律

- 测试保护当前核心能力，不测试口号。
- 单元测试保护独立规则。
- 集成测试保护真实接口或真实数据流。
- 端到端测试只在项目需要时启用。
- 不能把没有真实运行的链路写成已通过。

# Project Inception Skill

## 用途

当用户要新建项目、整理旧项目、搭项目骨架、确认产品需求、编写项目规则、建立开源协议或生产级验收边界时，先阅读并执行本文件。

本仓库是模板仓库。创建新项目时，优先照抄并改写根目录模板文件，而不是把规则重新发明一遍。

## 工作流

### 1. 需求分析

需求未确认时，不写业务代码。

先问清：

- 产品是什么。
- 用户是谁。
- 核心用户路径是什么。
- 当前必须做什么。
- 当前明确不做什么。
- 数据是什么、从哪里来、存在哪里。
- 是否需要账号、云端、本地离线、导入、导出。
- 是否涉及隐私、安全、密钥、成本或合规。
- 目标平台是什么：Web、桌面、小程序、App、CLI、服务端、库、脚本、文档项目等。
- 是否已有技术栈偏好或参考项目。
- 是否开源；未特别说明时默认 MIT。
- 怎样验证完成。
- 什么叫生产级完成。

不要一次问太多问题。优先问会影响架构和边界的问题。

### 2. 事实调查

如果目标目录已经存在，先调查再改。

必须检查：

- 当前目录结构。
- 是否已有 git 仓库。
- 是否已有 README、docs、脚本、测试、配置。
- 是否已有 `AGENTS.md`、`spec.md`、`plan.md`、`plan.example.md`。
- 是否已有 `.codex/skills/`。
- 当前技术栈是什么。
- 当前验证命令是什么。
- 哪些文件是当前事实。
- 哪些文件是运行产物。
- 哪些文件可能是私有配置或密钥。
- 当前工作区是否有未提交改动。

### 3. 项目骨架

需求确认后，按目标项目事实创建或整理：

- `AGENTS.md`
- `spec.md`
- `README.md`
- `LICENSE`
- `SECURITY.md`
- `CONTRIBUTING.md`
- `.gitignore`
- `plan.example.md`
- `.codex/skills/plan/SKILL.md`
- `.codex/skills/<project>-development/SKILL.md`

是否创建技术栈专属文件，必须由技术栈决定，不能硬套：

- `package.json`
- `pyproject.toml`
- `requirements.txt`
- `pom.xml`
- `build.gradle`
- `go.mod`
- `Cargo.toml`
- 小程序配置文件
- Docker 配置
- CI 配置
- 启动脚本

必须有项目级完整验证入口，但不强制载体。

## 硬规则

- 始终使用简体中文和项目 owner 交流。
- 先 research，再 plan，再实现。
- 用户输入是线索，不是结论。
- 当前仓库事实高于历史设计。
- 旧提交、旧文档、旧代码、测试绿灯、历史偏好都只是证据，不是绝对真理。
- 结论必须来自证据之间的一致性。
- 没有完成主链路事实调查，不动局部代码。
- 新事实推翻计划，先改计划，再继续。
- checklist 必须随进度更新。
- 禁止机械替换。
- 禁止跨文件批量正则替换。
- 禁止兼容转发。
- 禁止旧别名。
- 禁止历史适配层。
- 禁止旧语义包装。
- 禁止为了“平滑过渡”保留中间文件。
- 当前代码只服务当前生效实现。
- 不把旧能力、旧状态、旧数据写成当前产品事实。
- 历史只能作为 research 证据，不能进入当前产品主干。
- 不把未实现能力写进 `spec.md`、README、测试、CLI 输出或页面文案。
- 单一职责看变化原因，不看行数。
- 超过 300 行必须触发职责审查，但不是自动拆分理由。
- 能一句话说清职责、变化原因一致、内部耦合合理，可以保留。
- 状态管理、规则计算、数据读写、渲染展示、外部接线、错误处理、业务判断混在一起时，必须拆清边界。
- 不为了拆而拆。
- 不为了省事硬塞。
- 不交半成品。
- 交付必须闭环：research、设计、实现、测试、文档同步、验证、收口。
- commit / push 必须 owner 明确授权。
- 默认开源协议是 MIT。
- 不为了套模板制造无关章节。

## 为什么禁止历史兼容

快速演进项目的目标是尽快形成清晰、稳定、可验证的当前主干。

如果同时保留旧接口、旧字段、旧页面、旧目录、旧文案、旧数据形态和新实现，项目会变成两套事实并存：当前事实一套，历史事实一套，过渡逻辑一套，兼容分支一套。最后代码不是产品主干，而是历史包袱的垃圾场。

默认策略：

- 旧实现阻碍当前主干时，删除旧实现。
- 旧接口阻碍当前接口时，删除旧接口。
- 旧字段阻碍当前数据模型时，重建当前数据模型。
- 旧页面阻碍当前用户路径时，删除并重建当前页面。
- 旧测试保护旧行为时，删除并重写保护当前行为的测试。
- 旧文档描述旧事实时，删除或重写为当前事实。

## 历史文档规则

默认不创建历史文档，不主动建议写历史文档，不默认归档旧文档。

只有 owner 明确要求“写历史”“整理历史”“保留旧文档”“把旧内容做成 history.md”时，才创建或更新历史文档。

历史文档只用于接住上下文、记录试错轨迹、解释某条规则从哪个失败里来。它不是 `spec.md`，不是 README，不是产品宣传，不是当前实现规则。

## 模板文件

创建新项目时，按需复制并改写本仓库模板：

- `AGENTS.md`
- `spec.md`
- `plan.example.md`
- `CONTRIBUTING.md`
- `SECURITY.md`
- `LICENSE`
- `.gitignore`
- `.codex/skills/project-development/SKILL.md`
- `.codex/skills/plan/SKILL.md`

目标项目的 README 应面向用户和新读者，简短说明项目是什么、能做什么、怎么启动、怎么验证、使用什么协议。
